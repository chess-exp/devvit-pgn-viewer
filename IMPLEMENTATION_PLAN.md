# Devvit PGN Viewer Implementation Plan

## Goal

Build a Reddit Devvit Web app that creates an interactive custom post for one chess PGN game. The first release should let a moderator open a subreddit menu action, paste PGN text, and create a custom post that renders a chessboard, move list, metadata, and navigation controls inside Reddit.

## Plan Review

The original plan is directionally correct, but it needs these implementation changes before coding starts:

- Use the current Devvit Web React template shape: `src/client/*.html`, `src/client/*.tsx`, `src/server/routes/*`, `server.dir = "dist/server"`, and `server.entry = "index.cjs"`.
- Require Node 22.2+ for the React template. The latest Devvit CLI package also fails under Node 18 in this workspace.
- Store PGN in Redis by default. Devvit `postData` is only for a small pointer and summary payload, not the PGN body.
- Keep the MVP to manual moderator creation only. Do not configure the existing-post conversion menu item yet.
- Add a `forms` entry in `devvit.json`; menu-opened forms need a named form submission endpoint.
- Set custom post height through `submitCustomPost({ styles: { height: "TALL" } })`, not as an entrypoint field.
- Use `GET /api/pgn` from the webview with `context.postData`; do not accept an arbitrary `postId` query parameter for the MVP.

## MVP Decisions

- **Creator:** moderators only, through a subreddit menu action.
- **Posting identity:** app account by default. Do not request `reddit.asUser` permissions for the MVP.
- **Game count:** exactly one PGN game per post.
- **Storage:** Redis for PGN text; `postData` for the Redis key and compact metadata.
- **Client parsing:** parse PGN in the browser with `chess.js` to derive board positions and move list.
- **Server validation:** parse the same PGN server-side before writing Redis or creating a post.
- **Text fallback:** include the full PGN in Markdown code block form when within the MVP size limit.
- **Existing post conversion:** deferred until manual creation is stable.
- **Comments, multi-game PGNs, automation, and public directory listing:** deferred.

## Data Model

`postData` must stay compact.

```ts
export type PgnHeaders = {
  event?: string;
  site?: string;
  date?: string;
  round?: string;
  white?: string;
  black?: string;
  result?: string;
};

export type PgnPostData = {
  version: 1;
  kind: "pgn-viewer";
  redisKey: string;
  headers: PgnHeaders;
  plyCount: number;
  pgnLength: number;
  pgnSha256: string;
  createdAt: string;
};

export type RedisPgnRecord = {
  version: 1;
  pgn: string;
  headers: PgnHeaders;
  plyCount: number;
  pgnLength: number;
  pgnSha256: string;
  postId?: string;
  createdAt: string;
};
```

Redis key format:

```text
pgn-viewer:v1:pgn:{uuid}
```

Generate the UUID before creating the post. Save the Redis record first, use the generated key in `postData`, then update the Redis record with `postId` after `submitCustomPost` succeeds. If post creation fails, delete the Redis key.

## PGN Validation

Create `src/server/pgn.ts` with pure validation helpers:

- `normalizePgn(input: unknown): string`
- `validatePgn(input: unknown): ValidatedPgn | ValidationError`
- `extractHeaders(chess: Chess): PgnHeaders`
- `buildTextFallback(validated: ValidatedPgn): string`

Rules:

- Title is required, trimmed, and capped at 300 characters.
- PGN is required after trimming.
- MVP PGN length limit is `36_000` characters so the full PGN plus summary fits in `textFallback`.
- Parse with `new Chess().loadPgn(pgn, { strict: false })`.
- Reject PGNs with zero moves.
- Accept normal headers but do not require them.
- Treat comments, NAGs, castling, promotion, and checkmate as valid when `chess.js` accepts them.
- Display only the mainline move list in the MVP.
- Render all metadata and comments as React text, never raw HTML.

## Server Routes

### `POST /internal/menu/create-pgn-viewer`

Return `UiResponse.showForm`:

- `name: "createPgnViewer"`
- Fields:
  - `title`: `string`
  - `pgn`: `paragraph`
- Accept label: `Create`

### `POST /internal/form/create-pgn-viewer`

Flow:

1. Read `title` and `pgn` from the form body.
2. Validate `context.subredditName`.
3. Validate title and PGN.
4. Generate Redis key and save `RedisPgnRecord`.
5. Call `reddit.submitCustomPost` with:
   - `subredditName`
   - `title`
   - `entry: "default"`
   - compact `postData`
   - `textFallback: { text: buildTextFallback(validated) }`
   - `styles: { height: "TALL", backgroundColor: "#FFFFFFFF", backgroundColorDark: "#111111FF" }`
6. Update Redis record with the created `postId`.
7. Return `UiResponse.navigateTo` for the created post. If the generated types only accept a URL string, use the template's Reddit comments URL pattern.

On validation failure, return a clear error toast. On Redis write or post creation failure, clean up any partially written Redis record and return a generic failure toast.

### `GET /api/pgn`

Flow:

1. Read `context.postId` and `context.postData`.
2. Verify `postData.kind === "pgn-viewer"` and `version === 1`.
3. Read the Redis record by `postData.redisKey`.
4. Verify the Redis record hash/length match `postData`.
5. Return:

```ts
type PgnApiResponse =
  | {
      status: "ok";
      postId: string;
      pgn: string;
      headers: PgnHeaders;
      plyCount: number;
    }
  | {
      status: "error";
      message: string;
    };
```

Do not accept a caller-supplied post ID in the MVP.

## Client Implementation

Use `src/client/viewer.tsx` as the only MVP entrypoint.

Client flow:

1. Fetch `/api/pgn` on mount.
2. Show compact loading, error, and empty states that fit in an inline post.
3. Parse PGN with `chess.js`.
4. Build a `GameModel`:
   - headers
   - SAN move list
   - FEN position for ply `0..plyCount`
   - result
5. Keep `plyIndex` and `boardOrientation` in React state.
6. Render:
   - players/result header
   - responsive board
   - first/previous/next/last controls
   - flip-board control
   - scrollable mainline move list with active ply

Use `react-chessboard` approximately like:

```tsx
<Chessboard
  options={{
    position: currentFen,
    boardOrientation,
    allowDragging: false,
    showAnimations: true
  }}
/>
```

Use `lucide-react` icons for toolbar buttons, with accessible labels and tooltips.

## Layout Requirements

- Fit the primary inline experience within a `TALL` custom post.
- Board first on mobile; board plus move list side-by-side when width allows.
- Stable board dimensions using `aspect-ratio: 1 / 1`.
- Tap targets at least 40px.
- Avoid nested cards and heavy decorative layout.
- Avoid external image, font, analytics, or API requests.
- Use Reddit-compatible light/dark styling.
- Do not put instructional text in the app UI beyond validation/error messages.

## Definition Of Done

- Manual moderator-created PGN viewer posts work end to end in playtest.
- The app stores PGN in Redis and only compact metadata in `postData`.

## Open Questions

No question blocks implementation under the assumptions above. The recommended MVP answers are:

- **Who can create PGN viewer posts?** Recommended: moderators only for the first release. This keeps the first workflow easy to control, avoids spam/abuse problems from arbitrary large PGN submissions, and is simpler to validate in playtest. Later, regular users can be allowed after the app has rate limits, clearer validation messages, and moderator expectations are settled.
- **Who appears as the post author?** Recommended: the app account for the first release. Posting as the app account only needs normal Reddit permission and makes every generated viewer post clearly app-created. Posting as the initiating user requires Devvit user-action permissions, explicit user consent, and review-sensitive `userGeneratedContent` handling. That can be added later if attribution to the human submitter becomes important.

Choose different answers only if the first release must let ordinary subreddit members create viewer posts themselves, or if posts must appear under the human creator's Reddit username.
